# The Model Context Protocol (MCP): Architecture, Components, Transport, and Client-Server Communication

## Executive Summary

The Model Context Protocol (MCP) is an open protocol, announced and open-sourced by Anthropic on November 25, 2024, that standardizes how AI applications ("hosts") connect to external tools, data sources, and interaction templates via dedicated "servers" ([Anthropic](https://www.anthropic.com/news/model-context-protocol)). MCP follows a client-host-server architecture: a host application (e.g., Claude Code or Claude Desktop) spins up one dedicated client per server, and each client maintains a stateful, JSON-RPC 2.0-based session with its server over one of two standard transports â€” stdio for local subprocesses or Streamable HTTP for remote/networked servers ([MCP docs â€” Architecture](https://modelcontextprotocol.io/docs/learn/architecture)). The protocol splits cleanly into a data layer (message structure, lifecycle, and core primitives) and a transport layer (connection establishment, framing, authorization) ([MCP docs â€” Architecture](https://modelcontextprotocol.io/docs/learn/architecture)). Every connection begins with a capability-negotiation handshake (`initialize` â†’ server response â†’ `notifications/initialized`) before normal operation, and servers expose three core primitives â€” Tools, Resources, and Prompts â€” each independently discoverable and invocable through paired list/call JSON-RPC methods ([MCP Specification â€” Lifecycle, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle); [MCP Specification â€” Server Features, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/server/resources)). Distinct from static, in-band schema approaches like OpenAI function calling or stateless OpenAPI tool calls, MCP's stateful session model enables dynamic runtime discovery of tools across many distributed servers ([MarkTechPost](https://www.marktechpost.com/2025/10/08/model-context-protocol-mcp-vs-function-calling-vs-openapi-tools-when-to-use-each/)). The ecosystem has grown quickly: the official reference server repository has attracted roughly 88,000 GitHub stars ([GitHub â€” modelcontextprotocol/servers](https://github.com/modelcontextprotocol/servers)), with early adopters including Block, Apollo, Zed, Replit, Codeium, and Sourcegraph ([Anthropic](https://www.anthropic.com/news/model-context-protocol)).

## Key Findings

**Architecture and connection model**
- MCP follows a client-server architecture in which an MCP host (an AI application like Claude Code or Claude Desktop) creates one dedicated MCP client per MCP server â€” a strict 1:1 client-to-server relationship, with each client maintaining a dedicated connection to its corresponding server ([MCP docs â€” Architecture](https://modelcontextprotocol.io/docs/learn/architecture)).
- Local MCP servers using the stdio transport typically serve a single MCP client, while remote MCP servers using Streamable HTTP typically serve many MCP clients ([MCP docs â€” Architecture](https://modelcontextprotocol.io/docs/learn/architecture)).
- MCP is composed of two conceptual layers: an inner data layer implementing the JSON-RPC 2.0 exchange protocol (message structure, lifecycle management, and core primitives â€” tools, resources, prompts, notifications), and an outer transport layer handling connection establishment, message framing, and authorization ([MCP docs â€” Architecture](https://modelcontextprotocol.io/docs/learn/architecture)).
- Claude is configured as an MCP host/client through a JSON configuration file using an `mcpServers` object, where each entry specifies a launch command (e.g., `npx -y @modelcontextprotocol/server-filesystem /path/to/allowed/files`) and optional environment variables such as `GITHUB_PERSONAL_ACCESS_TOKEN` ([MCP docs â€” Example Servers](https://modelcontextprotocol.io/examples)).

**Transport mechanisms**
- MCP supports exactly two standard transports: stdio (standard input/output streams for local process communication with no network overhead) and Streamable HTTP (HTTP POST for client-to-server messages, with optional Server-Sent Events for streaming, supporting bearer tokens, API keys, custom headers, and OAuth) ([MCP docs â€” Architecture](https://modelcontextprotocol.io/docs/learn/architecture)).
- In stdio transport, the client launches the server as a subprocess; the server reads JSON-RPC messages from stdin and writes them to stdout, messages are newline-delimited and must not contain embedded newlines, and the server must not write anything to stdout other than valid MCP messages â€” stderr is reserved for optional UTF-8 logging ([MCP Specification â€” Transports, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports)).
- The Streamable HTTP transport, which replaced the older HTTP+SSE transport from protocol version 2024-11-05, requires the server to expose a single "MCP endpoint" (e.g., `https://example.com/mcp`) supporting both POST and GET, rather than separate SSE and message endpoints ([MCP Specification â€” Transports, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports)).
- For every client-to-server JSON-RPC request over Streamable HTTP, the client sends an HTTP POST with an `Accept` header listing both `application/json` and `text/event-stream`; the server responds with either `Content-Type: application/json` (a single JSON object) or `Content-Type: text/event-stream` to open an SSE stream that eventually carries the JSON-RPC response ([MCP Specification â€” Transports, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports)).
- A Streamable HTTP server may assign a session ID during initialization via an `MCP-Session-Id` header on the response carrying the `InitializeResult`; the session ID must contain only visible ASCII characters (0x21â€“0x7E), the client must include it on all subsequent requests, and the server responds `404 Not Found` once the session is terminated ([MCP Specification â€” Transports, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports)).
- Clients can explicitly terminate an HTTP session by issuing an HTTP DELETE with the `MCP-Session-Id` header, to which the server may respond `405` if client-initiated termination isn't supported ([MCP Specification â€” Lifecycle, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle)).

**Session lifecycle and version negotiation**
- MCP defines a three-phase connection lifecycle: (1) Initialization â€” capability negotiation and protocol version agreement, (2) Operation â€” normal protocol communication, and (3) Shutdown â€” graceful termination ([MCP Specification â€” Lifecycle](https://modelcontextprotocol.io/specification/2025-03-26/basic/lifecycle)).
- The client must initiate the initialization phase by sending an `initialize` request (JSON-RPC method `"initialize"`) containing its supported `protocolVersion`, `capabilities` (e.g., `roots`, `sampling`, `elicitation`, `tasks`), and `clientInfo`; the `initialize` request must not be part of a JSON-RPC batch ([MCP Specification â€” Lifecycle, 2025-03-26](https://modelcontextprotocol.io/specification/2025-03-26/basic/lifecycle); [MCP Specification â€” Lifecycle, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle)).
- The server responds with its own `protocolVersion`, `capabilities` (e.g., `logging`, `prompts`, `resources`, `tools`, `tasks`), `serverInfo`, and optional `instructions` ([MCP Specification â€” Lifecycle, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle)).
- After receiving the server's response, the client must send a `notifications/initialized` notification before normal operation begins; before this handshake completes, clients and servers should not send non-ping (and, for servers, non-logging) requests ([MCP Specification â€” Lifecycle, 2025-03-26](https://modelcontextprotocol.io/specification/2025-03-26/basic/lifecycle); [MCP Specification â€” Lifecycle, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle)).
- Protocol version negotiation is asymmetric: if the server supports the client's requested `protocolVersion`, it must echo that exact version back; otherwise it must respond with another version it supports (preferably its latest). If the client does not support the version the server returns, it should disconnect ([MCP Specification â€” Lifecycle, 2025-03-26](https://modelcontextprotocol.io/specification/2025-03-26/basic/lifecycle); [MCP Specification â€” Lifecycle, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle)).
- For HTTP transports, the client must also include an `MCP-Protocol-Version: <version>` header on all requests following initialization ([MCP Specification â€” Lifecycle, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle)).
- Shutdown over stdio: the client should first close the server's input stream, then wait for it to exit or send SIGTERM if it doesn't, and finally send SIGKILL if it still hasn't exited; the server may also initiate shutdown by closing its output stream and exiting. For HTTP transports, shutdown is simply closing the associated connection(s), since no dedicated shutdown messages are defined ([MCP Specification â€” Lifecycle, 2025-03-26](https://modelcontextprotocol.io/specification/2025-03-26/basic/lifecycle); [MCP Specification â€” Lifecycle, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle)).

**Message format (JSON-RPC 2.0)**
- All MCP messages follow JSON-RPC 2.0 and fall into three types: Requests (must include a non-null, non-reused string/integer `id`), Responses (must include the same `id` as the corresponding request, and must set either `result` or `error` but not both), and Notifications (must not include an `id`, and the receiver must not respond to them). Implementations may send JSON-RPC batches but must support receiving them ([MCP Specification â€” Base Protocol Overview](https://modelcontextprotocol.io/specification/2025-03-26/basic)).

**Server-offered primitives**
- MCP defines three server-offered primitives with distinct control models: Tools are model-controlled functions the AI model can invoke (e.g., calling APIs, running computations); Resources are application-driven data (files, database schemas, etc.) that clients decide how to surface; and Prompts are user-controlled templated messages/workflows explicitly selected by the user (e.g., exposed as slash commands) ([MCP docs â€” Server Concepts](https://modelcontextprotocol.io/docs/learn/server-concepts)).
- Each primitive is discoverable via a `*/list` method (e.g., `tools/list`) and, for tools, invoked via `tools/call` ([MCP docs â€” Architecture](https://modelcontextprotocol.io/docs/learn/architecture)).
- Each primitive exposes exactly two core JSON-RPC methods plus a list-changed notification: tools use `tools/list` and `tools/call` with `notifications/tools/list_changed`; resources use `resources/list` and `resources/read` (plus `resources/templates/list`, `resources/subscribe`, and update notification `notifications/resources/updated`) with `notifications/resources/list_changed`; prompts use `prompts/list` and `prompts/get` with `notifications/prompts/list_changed` ([MCP Specification â€” Server Features, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/server/resources)).
- Tool execution failures and protocol-level errors are handled differently: business/API/validation failures during a tool call are returned inside a successful JSON-RPC result with `"isError": true` in the `tools/call` response (so the LLM can see and self-correct), whereas malformed requests or unknown tool names return a standard JSON-RPC error object (e.g., code `-32602`) instead of a result ([MCP Specification â€” Server Features: Tools, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/server/tools)).

**Client-offered primitives**
- MCP also defines primitives that clients expose back to servers: Sampling (servers request LLM completions from the client's AI application via `sampling/createMessage`), Elicitation (servers request additional user input via `elicitation/create`), and Logging (servers send log messages to clients) ([MCP docs â€” Architecture](https://modelcontextprotocol.io/docs/learn/architecture)).

**Comparison to alternative approaches**
- Unlike OpenAI function calling, which relies on static schema definitions provided upfront in-band with each LLM API call (no separate session lifecycle), and OpenAPI Tools, which are stateless per-operation HTTP(S) requests, MCP uses a stateful JSON-RPC session over stdio or HTTP-based transports with dynamic runtime tool discovery via `tools/list`, allowing a host to enumerate and route to tools across multiple distributed servers ([MarkTechPost](https://www.marktechpost.com/2025/10/08/model-context-protocol-mcp-vs-function-calling-vs-openapi-tools-when-to-use-each/)).

**Origins and ecosystem**
- Anthropic announced and open-sourced MCP on November 25, 2024, releasing the specification, SDKs, local MCP server support in Claude Desktop, and pre-built servers for Google Drive, Slack, GitHub, Git, Postgres, and Puppeteer; early adopters cited include Block and Apollo, with development-tool companies Zed, Replit, Codeium, and Sourcegraph also integrating MCP ([Anthropic](https://www.anthropic.com/news/model-context-protocol)).
- The official `modelcontextprotocol/servers` repository lists seven current reference implementations demonstrating core MCP features and SDK usage: Everything (test server with prompts, resources, and tools), Fetch (web content fetching), Filesystem (secure file operations with configurable access controls), Git (repository read/search/manipulate tools), Memory (knowledge-graph-based persistent memory), Sequential Thinking (dynamic problem-solving via thought sequences), and Time (timezone conversion) ([MCP docs â€” Example Servers](https://modelcontextprotocol.io/examples)).
- As of the fetch date, the `modelcontextprotocol/servers` repository had accumulated approximately 88,000 stars and 11,100 forks, with 358 open issues and 261 open pull requests, reflecting large-scale community engagement ([GitHub â€” modelcontextprotocol/servers](https://github.com/modelcontextprotocol/servers)).

## Open Questions

- The claims include two versions of the specification (2025-03-26 and 2025-11-25) with overlapping but not fully reconciled lifecycle and transport details (e.g., `tasks` capability and `MCP-Protocol-Version` header appear only in the 2025-11-25 claims). It is unclear from the provided material exactly which behaviors are new in 2025-11-25 versus carried over unchanged from 2025-03-26, or what other changes the version bump introduced beyond those documented here.
- The claims do not specify how authorization/OAuth flows concretely work end-to-end for Streamable HTTP servers beyond noting that bearer tokens, API keys, custom headers, and OAuth are supported/recommended.
- The claims do not detail how a host aggregates or deduplicates tools/resources/prompts when connected to multiple servers simultaneously, nor how naming collisions across servers are resolved.
- The GitHub star/fork/issue counts are presented as "as of the fetch date" without that date being specified in the claim itself, so their currency relative to this report's date is uncertain.
- The claims do not describe error handling or retry semantics for transport-level failures (e.g., dropped HTTP connections mid-stream) beyond basic shutdown procedures.
